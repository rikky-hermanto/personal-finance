## Objective

Write unit tests for `TransactionService` — the only service without test coverage. Covers transaction persistence, duplicate detection logic, balance calculation, and single-record lookup.

## Acceptance Criteria

- [ ] `AddTransactionsAsync_WithValidDtos_PersistsAndReturnsMappedDtos`
- [ ] `AddTransactionsAsync_WhenValidationFails_ThrowsInvalidOperationException`
- [ ] `FilterOutDuplicatesAsync_WithNoDuplicates_ReturnsAllTransactions`
- [ ] `FilterOutDuplicatesAsync_WithExistingDuplicate_FiltersItOut`
- [ ] `FilterOutDuplicatesAsync_DuplicateKeyIsCompositeOf_Date_Description_Flow_Type_Wallet`
- [ ] `GetTransactionsWithBalanceAsync_CalculatesRunningBalance_CrAddsDbSubtracts`
- [ ] `GetTransactionsWithBalanceAsync_WithWalletFilter_ReturnsOnlyMatchingWallet`
- [ ] `GetTransactionByIdAsync_WithExistingId_ReturnsDto`
- [ ] `GetTransactionByIdAsync_WithNonExistingId_ReturnsNull`
- [ ] All tests use `UseInMemoryDatabase` + `IDisposable`
- [ ] `dotnet test` passes

## Technical Notes

- Use real `AppDbContext` (in-memory) and mock `IMediator`
- For `AddTransactionsAsync`, mock mediator's `Send` to return a `Transaction` entity
- Balance calculation test: seed 3 transactions (2 CR, 1 DB), verify running balance at each position
- Duplicate detection: seed 1 existing transaction, submit 2 (1 duplicate + 1 new), verify only 1 returned
- File location: `tests/PersonalFinance.Tests/Services/TransactionServiceTests.cs`

## Dependencies

- None

## Notes

_`FilterOutDuplicatesAsync` uses a composite key hash — the exact key format `{Date:u}|{Description}|{Flow}|{Type}|{Wallet}` must be tested explicitly to prevent regression._
