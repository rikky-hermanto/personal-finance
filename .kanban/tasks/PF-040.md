## Objective

Fix `GET /api/transactions/aggregated`: the 6-month cash flow window is always computed from `DateTime.Now`, ignoring the `year` and `month` query parameters. Viewing a historical month returns incorrect cash flow context.

## Bug Details

**File:** `api/src/PersonalFinance.Api/Controllers/TransactionsController.cs` lines 233–248

**Current code (broken):**
```csharp
for (int i = 5; i >= 0; i--)
{
    var targetDate = DateTime.Now.AddMonths(-i);  // always anchored to today
    ...
}
```

**Example:** Requesting `?year=2024&month=6` returns the Summary/CurrentMonth stats for June 2024 correctly, but the CashFlow section always shows the 6 months ending at today (e.g., Sep 2025–Feb 2026), not around June 2024.

**Expected:** CashFlow should show the 6 months ending at the requested `year`/`month`.

## Acceptance Criteria

- [ ] Cash flow window anchored to the requested `year`/`month` (defaulting to current date when not provided)
- [ ] `GET /api/transactions/aggregated?year=2024&month=6` returns CashFlow for Jan 2024–Jun 2024
- [ ] `GET /api/transactions/aggregated` (no params) still returns the last 6 months from today
- [ ] `dotnet build` succeeds
- [ ] Integration test added verifying cash flow window for a historical month

## Technical Notes

**Fix:**
```csharp
var anchorDate = new DateTime(currentYear, currentMonth, 1);
for (int i = 5; i >= 0; i--)
{
    var targetDate = anchorDate.AddMonths(-i);
    ...
}
```

Also: `CurrentMonth.Month` field currently uses `DateTime.Now.ToString("MMMM yyyy")` — should use `new DateTime(currentYear, currentMonth, 1).ToString("MMMM yyyy")` for consistency.
