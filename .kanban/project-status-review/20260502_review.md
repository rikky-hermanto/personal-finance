## Top Priorities — Report

Ranked. Each entry includes scope, why now, blast radius, and what blocks/unblocks.

### 1. PF-S08 — Supabase Auth (JWT middleware + user_id columns + RLS) — [#71](https://github.com/rikky-hermanto/personal-finance/issues/71)

- **Why now:** This is the documented next step in the Supabase migration sequence (Phase 3). Phases 4–6 (Storage, Webhooks, Realtime, RAG) all depend on JWT-based RLS being in place — without `auth.uid()` to enforce isolation, the storage bucket and webhook flow can't ship safely.
- **Scope:** JWT bearer middleware in `Program.cs`, `user_id uuid` columns on `transactions` and `category_rules`, RLS policies, service-role vs anon client wiring.
- **Blast radius:** API contract change — every handler needs the user's id from claims. Touches all 5 entity handlers.
- **Blocks:** PF-S09, PF-S10, PF-S11, PF-S12, PF-S13.

### 2. PF-S09 — Frontend Supabase Auth (login/signup + JWT forwarding) — [#72](https://github.com/rikky-hermanto/personal-finance/issues/72)

- **Why now:** Pairs naturally with PF-S08 — backend JWT validation is useless without frontend issuing the token. Doing them in one cycle avoids a half-broken state.
- **Scope:** `@supabase/supabase-js` client, login/signup pages, `Authorization: Bearer ${token}` injection in `src/api/*.ts` fetch calls.
- **Blast radius:** New pages + every API client function gains an `Authorization` header.
- **Blocks:** End-to-end usable app post-migration.

### 3. PF-051 — Add ILogger to all services, handlers, parsers — [#60](https://github.com/rikky-hermanto/personal-finance/issues/60)

- **Why now:** Zero `ILogger` usage across the .NET codebase today (governance ERR-02). Right before the surface area expands again (Auth, Storage, Webhooks), adding structured logging gives observability into the new flows for free instead of retrofitting later. Also low-risk and parallelizable with Auth work.
- **Scope:** Inject `ILogger<T>` into ~15 classes, replace `Console.WriteLine` in `NeoBankPdfParser`, log at handler boundaries and on errors.
- **Blast radius:** Constructor signatures change; mostly mechanical.
- **Blocks:** Nothing, but unblocks meaningful debugging of upcoming async/event-driven flows.

### 4. PF-S10 — Supabase Storage (bank-statements bucket + StorageService + upload endpoint) — [#73](https://github.com/rikky-hermanto/personal-finance/issues/73)

- **Why now:** Direct successor to Auth — the storage bucket needs RLS policies that reference `auth.uid()`. Also the gateway to the event-driven AI pipeline (PF-S11), so picking it up immediately after Auth keeps momentum on the migration's "value moment" (PDF upload → AI extraction).
- **Scope:** Bucket creation migration, `IStorageService` + Supabase Storage implementation, upload endpoint that writes to `statement_uploads` table.
- **Blast radius:** New endpoint + new service. Replaces current direct-upload-to-parser path for PDFs.
- **Blocks:** PF-S11, PF-S12.

### 5. Choose one of these as #5 — different strategic bets:

**5a. PF-043 — Wise CSV Parser + FX conversion — [#52](https://github.com/rikky-hermanto/personal-finance/issues/52)**
- **Bet:** Bank coverage. Wise is one of the 5 target banks; pure CSV (deterministic, zero LLM cost). Independent of the Supabase migration — can ship alongside it.
- **Effort:** Small. Pattern is already established by `BcaCsvParser`. New element is FX rate → IDR conversion.

**5b. PF-S11 — Event-driven AI pipeline (Webhook → Python /webhooks/process) — [#74](https://github.com/rikky-hermanto/personal-finance/issues/74)**
- **Bet:** Stay on the migration critical path. After Storage, this is the moment LLM extraction becomes real for users.
- **Effort:** Larger. Requires Database Webhook config, webhook secret validation, and `supabase-py` write-back.

**5c. PF-052 — TypeScript strict mode + `Transaction.id` type fix — [#61](https://github.com/rikky-hermanto/personal-finance/issues/61)**
- **Bet:** Pre-existing bug catch. CLAUDE.md flags `id: string` vs API returning `number` as a known type mismatch. Strict mode would have caught this. Cheap and prevents recurring frontend bugs.
- **Effort:** Small-to-medium. May surface other latent type issues.

**5d. New Issue #77 — LLM Tracing for AI Service**
- **Bet:** Observability. Almost certainly overlaps with PF-024 (AI Observability + Cost Tracking). Probably should be merged or one closed.
- **Effort:** Unclear — issue is research-only at the moment. Recommend folding into PF-024 before scheduling.
